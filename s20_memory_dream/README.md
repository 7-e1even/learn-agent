# s20 · 自动长期记忆（dream 固化）

让 agent 在几十个项目、几百个会话之间自动沉淀该记住的事——又不变成账单，也不变成垃圾堆。

## 问题

s17 的复盘环解决的是"这次对话学到了什么"；这一章的问题更大一号：**跨会话、跨项目的长期记忆怎么自动维护**。你在项目 A 告诉过 agent"这里用 pnpm"，三天后在项目 A 开新会话，它应该已经知道。

手动记（用户自己写 AGENTS.md）留不下对话里的东西；让模型"随手记"（prompt 里指示它觉得重要就写笔记）靠自觉，经常忘。而全自动的那一端是另一个坑：每个 turn 结束都抽取一次，账单直接乘上 turn 数。

更隐蔽的坑有三个，全部来自真实产品的踩坑记录：

- **时机**：正在进行到一半的工作被总结成"事实"——迁移做到一半时固化，笔记本里就写下了"项目用 v1 兼容层"，八小时后真相反转。
- **选材**：凡是"看起来持久"的都写下来（write-ahead），几个月后笔记本 80% 是从未被用到的死重，每轮还都注入 prompt。
- **只进不出**：项目决策会变，被推翻的旧笔记一直躺在索引里，模型新旧各信一半。

这一章的原型（Reina 的第一版实现）本身就是反面教材：门槛一到就往**聊天里弹一条"25 个会话已积累，建议固化"的气泡**，骑在用户的回合上执行。用户的原话是"我项目一多，这个功能就是累赘"。重做后的形状是本章的正题。

## 解决方案

骨架来自 grok 的 memory dream（session 结束后台跑一次固化，写前先打锁戳），三个关键补丁各有出处：idle 过滤来自 codex（`min_rollout_idle_hours`：只总结已经冷却的会话）、修剪与"小而准"来自 claude-code 的 dream 巩固、**检索即信号**是原创（调研过的九家实现都没有）。

拧成一句话：**inline 随手记是主通道，检索命中历史时提示升级是需求驱动的第二通道，后台 dream 是兜底——三条写入通道全部由需求驱动，没有一次"专门为记忆而生的高频模型调用"；配一个 delete 出口让记忆敢于自动生长。**

**它和 s17 复盘环的分工**（两者共用 fork 基建，但不是一回事，也不该合并）：

| | s17 复盘环 | s20 dream 固化 |
|---|---|---|
| 回答的问题 | "刚才**这段对话**学到了什么" | "最近**这批会话**里什么值得长期记住" |
| 材料 | 当前会话完整 transcript（深读） | 其他会话的简报——标题+摘要（广扫） |
| 触发 | 同会话内每 N 轮 | 跨会话：冷却 + 会话计数 + idle |
| 产出 | 全局笔记 + 技能文件 | 只写全局笔记 + **删**被推翻的旧笔记 |
| 默认 | 关（聊得久就反复跑，贵） | 开（最多几天一次，便宜） |

打个比方：复盘是**当天写日记**（趁热，有全文深度，能沉淀技巧成技能），固化是**定期整理笔记本**（跨几十个会话，誊正册 + 撕旧页）。不合并的原因是两者的强项恰是对方的盲区——复盘只看得见自己这一个会话，固化只拿得到摘要级材料；而且整个记忆系统里**只有固化负责"删"**，复盘环只会往里加。

## 运行

演示不需要 API key：

```sh
node s20_memory_dream/demo.mjs
```

四笔账，真实运行输出（节选）：

```
━━━ 账一：什么时候写 —— 同一个月（3 项目 · 107 会话 · 373 turns），四种策略的后台调用数 ━━━
  A 每 turn 抽取（claude-code）： 373 次    —— 最细也最贵，靠 inline 互斥 + 节流才压得住
  B 启动扫描（codex）        ：  29 次    —— 摊在每天首启，永不和活跃会话抢资源
  C session 结束（grok）     ：  20 次    —— 依赖干净的「会话结束」信号（CLI 才有）
  D turn 末门槛（Reina）     ：   9 次    —— 检查零成本（两条 SQL），开跑按项目批量结账

━━━ 账二：idle 过滤 ━━━
  不过滤：❌ 半截事实入库：下个会话会信「项目用 v1」，而真相 8 小时后就反转了
  闲置≥6h：项目已全量迁到 v2 / CI 用 pnpm，node 22

━━━ 账三：检索即信号 ━━━
  write-ahead：写 20 条，其中 16 条从未被再次用到 —— 笔记本 80% 是死重
  检索即信号：只有被再次查找的 4 条被提示升级

━━━ 账四：修剪出口 —— 10 次固化后的笔记本 ━━━
  只进不出：30 条，其中 10 条已被推翻但仍在索引里
  有修剪  ：20 条，全部当前有效
```

## 实现

### ① 什么时候写：检查和执行分开计价

账一的四种策略差两个数量级，但真正的设计洞察不在"选哪档频率"，而在**把"要不要固化"的检查和"固化"本身分开计价**。检查做成纯数据库查询（两条 SQL：距上次固化多久了、攒了几个新会话），零 token，每个 turn 结束都跑得起；执行才是模型调用，被三道门槛压住：总开关、每项目 72h 冷却、攒够 10 个闲置会话。

为什么挂在 turn 末而不是 session 结束（grok 的选择）？桌面应用没有干净的"会话结束"信号——用户就是把窗口一直开着。turn 边界是桌面场景里唯一可靠的钩子，重门槛把实际频率压到"每项目隔几天一次"。

另一个 grok 细节值得原样保留：**先打时间戳再执行**（戳即锁）。同一工作区并发两个会话同时到达门槛时，第一个把 `consolidated_at` 戳下去，第二个的检查立即不过——不需要真正的锁文件。跑挂了也不回滚：等下个冷却周期重试，比崩溃恢复逻辑便宜得多。

### ② idle 过滤：只总结尘埃落定的工作

codex 的 `min_rollout_idle_hours` 是这一章最划算的移植（一个 SQL 上界参数）：**只有闲置 ≥6h 的会话才计入门槛、才进固化材料**。道理是账二演示的：记忆是给未来会话用的，晚 6 小时写没有任何损失；提前写的代价是把中间态固化成"事实"。

有个必须同步的细节：门槛计数和材料构建**必须用同一条截止线**。如果计数不带 idle 过滤而材料带，会出现"门开了、材料却被滤空"的空转循环。附带效果：当前会话永远是"刚更新"的，天然不计入门槛——如果你的测试之前靠当前会话凑够第 N 个，会在这里翻车（真实踩坑）。

### ③ 检索即信号：什么值得写，让需求投票

这一章的原创部分。问题：怎么知道一个事实"值得"进长期记忆？write-ahead 的答案是"看起来持久就写"，账三演示了下场——80% 死重。更好的判据是：**当模型在新会话里被迫重新查找一个旧会话已经知道的事实时，这个事实的持久价值刚刚被现实投了一票。**

实现小到不像一个功能：跨会话搜索工具（`search_memory`）的结果里如果有来自**历史会话**的命中，就在输出尾部附一行——"这些命中来自过去的会话，是你不得不再查一遍的知识；如果它以后还会被用到，现在就把它折进全局笔记本"。零调度器、零额外模型调用，写入时机完全由真实需求驱动。本会话的命中不触发（还没证明跨会话价值）。

### ④ 修剪与防漂移：记忆的出口

只进不出的记忆最终反噬（账四）。两道保险，一写一读：

- **写侧**：固化代理的工具白名单里放进 `delete_note`，指令里加一条职责——被新会话推翻/取代的旧笔记顺手删掉。轻量版不需要"使用次数追踪"字段；靠固化代理对比新材料与旧索引就能判断谁被推翻了。
- **读侧**：笔记索引注入 prompt 的段落末尾加一句防漂移（claude-code 的教训原句是 "the memory says X exists is not the same as X exists now"）：笔记反映写入时的状态，引用其中的文件/flag/命令前先核实仍存在——不对就修笔记。

### 接进真实 agent

固化跑在一个**分离的受限子引擎**里，完整复用 s17 的基建：turn 末钩子 `maybeConsolidateMemory()` 检查门槛，到点 fire-and-forget 起 fork（`.catch(reportBackgroundError)`，永不阻塞当前轮），三条守卫同款——`isSubagent` 挡递归、开关、`inFlight` 防叠跑。fork 用专属 profile：自动批准模式 + **工具白名单只有笔记五件套**（note/update/read/list/delete）——没有 shell、没有网络、没有文件写，白名单就是全部安全故事。材料是会话简报（标题+摘要），不是全量 transcript——固化要的是"发生过什么"的目录，不是逐字回放。

## 练习

1. 账一的策略 C（session 结束触发）在桌面应用里退化，因为窗口常开、没有结束信号。设计一个
   替代信号：**工作区切换**——用户从项目 A 切到项目 B 的瞬间，A 的所有会话天然全部转入闲置。
   写出这个触发器的伪代码，并回答：它和 turn 末门槛检查是替代关系还是叠加关系？切换后立即
   固化和等 idle 满 6h 再固化，哪个对（提示：想想切回来又切走的抖动）。
2. ④的修剪是"轻量版"：没有使用追踪，靠固化代理判断谁被推翻。升级版是给每条笔记加
   `lastRecalledAt`（被 read_note / 搜索命中时更新），固化时归档 30 天未用的。现在回答一个
   陷阱题：笔记索引每轮都注入 prompt——"被注入"算不算"被使用"？如果算，会发生什么？
   （提示：一切永远新鲜，追踪失效。想清楚哪些事件才配更新这个时间戳。）

## 与真实产品对照（延伸阅读）

2026-07 对九家实现的横向核验（全部读了源码）：真·后台自动记忆只有四家——**claude-code**（最全：每 turn 后台抽取 + 24h/5 会话 dream 巡检 + 每 turn 小模型选 top-5 相关记忆注入，一事一 md + MEMORY.md 索引限 200 行）、**codex**（最重：启动时双阶段管线，逐 rollout 抽取→全局合并，SQLite 任务租约 + 记忆目录本身是 git 仓库、靠 diff 做增量遗忘，gate 六七道）、**grok**（本章骨架）、**hermes**（生命周期钩子矩阵 + 可插拔 mem0 等外部后端）。cherry-studio / deepagents / Kun 是"inline 工具 + prompt 指引"的半自动；opencode / kimi-code / pi 干脆没有。

Reina 的对照结论延续 s17 的"移植核心、砍外围"：骨架取 grok（最简的完整形态），补 codex 的 idle 过滤和 claude-code 的修剪+防漂移，各家都没有的检索即信号自己补上。**否决清单**同样有信息量：否决 claude-code 的每 turn 抽取（成本最高档，检索即信号以零成本覆盖其高价值子集）、否决 codex 的额度 gate（BYOK 没有后端额度概念）和 git-diff 遗忘（重型基建，delete_note 够用）、否决 hermes 的压缩前抽取钩子（Reina 的压缩 offload 已无损保留被压内容——如果你的 s06 压缩是有损的，这个钩子才值得抄）。
